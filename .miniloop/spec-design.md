# Design: `tonic install`

## CLI Surface

### New subcommands

```
tonic install <source> [--copy] [--force]
tonic uninstall <name>
tonic installed
```

**`tonic install <source>`**

| Source form | Example | Behavior |
|---|---|---|
| Local path | `tonic install .` or `tonic install ../tonic-loops` | Symlink project dir into `~/.tonic/packages/<name>/` (editable install). `--copy` snapshots instead. |
| Git URL | `tonic install https://github.com/user/tonic-loops.git` | Clone into `~/.tonic/packages/<name>/`. |
| Git URL + ref | `tonic install https://github.com/user/tonic-loops.git#v1.2` | Clone and checkout the specified ref. |
| Registry name | `tonic install tonic-loops` | **Future.** Error with "registry install not yet supported; use a path or git URL". |

Flags:
- `--copy` — force copy instead of symlink for local-path installs
- `--force` — overwrite existing shims owned by another package

**`tonic uninstall <name>`** — removes shims from `~/.tonic/bin/`, removes entry from `packages.toml`, optionally removes cached source from `~/.tonic/packages/<name>/`.

**`tonic installed`** — lists installed packages with source, installed binaries, and install date. Uses a separate verb (not `install --list`) to keep `install` focused on the install action.

### Handler pattern

New file: `src/cmd_install.rs`, following the existing `cmd_*.rs` pattern:
- `handle_install(args: Vec<String>) -> i32`
- `handle_uninstall(args: Vec<String>) -> i32`
- `handle_installed(args: Vec<String>) -> i32`

Dispatch added to `run()` in `src/main.rs`:
```rust
Some("install") => handle_install(iter.collect()),
Some("uninstall") => handle_uninstall(iter.collect()),
Some("installed") => handle_installed(iter.collect()),
```

## Directory Layout

```
~/.tonic/
├── bin/                      # User adds to PATH; contains shims
│   ├── miniloops             # Shell shim → tonic run ~/.tonic/packages/tonic-loops ...
│   └── my-tool
├── packages/                 # Cached project source trees
│   ├── tonic-loops/          # Symlink (local) or clone (git)
│   │   ├── tonic.toml
│   │   ├── src/
│   │   └── bin/
│   │       └── miniloops     # Original bin script (discovery source)
│   └── my-tool/
└── packages.toml             # Global install manifest
```

`~/.tonic/` is created on first `tonic install` invocation. No prior global state exists — this is a clean addition.

## Shim Generation

### Discovery

When installing a package, `tonic install` discovers binaries in this order:

1. **`bin/` directory** — scan for executable files. Each file becomes a shim. The shim name matches the filename (without extension if `.sh`).
2. **Fallback** — if no `bin/` directory exists but `tonic.toml` has `[package] name = "foo"`, create a single shim named `foo` that runs `tonic run <package-dir>`.
3. **No binaries found** — error: "no installable binaries found; add executables to bin/ or declare [package] name".

### Shim template

```sh
#!/bin/sh
# Generated by tonic install — do not edit
# Package: <name>
# Source: <source-type>
set -eu
exec tonic run '<abs-path-to-cached-package>' "$@"
```

Key details:
- Uses `exec` to replace the shell process (no extra PID).
- Quotes the package path to handle spaces.
- The `tonic run` invocation matches the existing pattern in `tonic-loops/bin/miniloops`.
- Shims are `chmod +x` on creation.

### Per-binary shims

If a `bin/` script takes a subcommand name (like `bin/miniloops` which receives args and delegates to `tonic run "$REPO_DIR" "$@"`), the generated shim replicates this by pointing at the cached package root. The original `bin/miniloops` script inside the package still works — but the installed shim in `~/.tonic/bin/` is a simpler wrapper since the package path is known at install time.

For `bin/` entries that are subcommand-specific (e.g., a hypothetical `bin/test` that runs `tonic test`), the shim delegates to the original script rather than reimplementing its logic:

```sh
#!/bin/sh
# Generated by tonic install — do not edit
# Package: <name>, bin: test
set -eu
exec '<abs-path-to-cached-package>/bin/test' "$@"
```

**Decision: two shim modes.**
- If `bin/<name>` is a shell script that invokes `tonic run`, generate a simplified shim that calls `tonic run` directly with the resolved package path.
- Otherwise, generate a passthrough shim that `exec`s the original `bin/<name>` script.

For MVP, **use passthrough shims exclusively** — they're correct for all cases and avoid parsing the original script's intent. Optimization (direct `tonic run` shims) can come later.

## Global Manifest: `~/.tonic/packages.toml`

```toml
# Managed by `tonic install` — manual edits are preserved but may be overwritten.

[packages.tonic-loops]
source = "path"
path = "/Users/rook/projects/tonic-loops"
symlink = true
bins = ["miniloops"]
installed_at = "2026-03-27T12:00:00Z"

[packages.other-tool]
source = "git"
url = "https://github.com/user/other-tool.git"
ref = "main"
commit = "abc1234def5678"
bins = ["other-tool"]
installed_at = "2026-03-27T14:00:00Z"
```

Parsed via `toml` crate (already a dependency). New struct:

```rust
struct InstalledPackage {
    source: InstallSource,  // Path { path, symlink } | Git { url, ref, commit }
    bins: Vec<String>,
    installed_at: String,
}
```

## Install Flow (step by step)

```
tonic install ../tonic-loops
```

1. **Resolve source** — canonicalize `../tonic-loops` to absolute path.
2. **Load target manifest** — read `tonic.toml` from the source to get package name. If no `[package] name`, derive from directory name.
3. **Check for conflicts** — if package name already installed, error (unless `--force`). If any bin name conflicts with another package's bins, error (unless `--force`).
4. **Ensure `~/.tonic/` structure** — create `~/.tonic/bin/`, `~/.tonic/packages/` if missing.
5. **Link/copy source** — symlink `~/.tonic/packages/<name>` → source path (or copy with `--copy`). For git: clone and checkout.
6. **Sync dependencies** — if the package has `[dependencies]`, run the equivalent of `tonic deps sync` in the cached package directory.
7. **Discover binaries** — scan `bin/` directory (or fall back to package name).
8. **Generate shims** — write passthrough shim scripts to `~/.tonic/bin/<bin-name>`, `chmod +x`.
9. **Update `packages.toml`** — record the install.
10. **Print summary** — list installed binaries. On first install, print PATH setup instructions.

## Uninstall Flow

```
tonic uninstall tonic-loops
```

1. Load `packages.toml`, find entry.
2. Remove each shim from `~/.tonic/bin/`.
3. Remove `~/.tonic/packages/<name>/` (remove symlink or delete cloned tree).
4. Remove entry from `packages.toml`.
5. Print confirmation.

## PATH Setup UX

On first `tonic install` (when `~/.tonic/bin/` is first created), print:

```
To use installed commands, add ~/.tonic/bin to your PATH:

  # bash/zsh
  export PATH="$HOME/.tonic/bin:$PATH"

  # fish
  fish_add_path ~/.tonic/bin

Add this to your shell profile to make it permanent.
```

No automatic shell profile modification (matches Cargo's approach, avoids surprising the user).

## Conflict Resolution

- **Same package reinstall**: `tonic install` on an already-installed package updates it (re-symlink/re-clone, regenerate shims).
- **Binary name conflict across packages**: error with message "binary 'foo' already installed by package 'bar'; use --force to overwrite".
- **`--force`**: overwrites shims and updates `packages.toml` to reassign the binary to the new package.

## Error Handling

All errors use `CliDiagnostic` with actionable hints, matching existing patterns:
- Missing `tonic.toml` in target: "path does not appear to be a tonic project (no tonic.toml found)"
- No binaries found: "no installable binaries found in <path>; add executables to bin/"
- Binary conflict: "binary '<name>' already installed by package '<pkg>'; use --force to overwrite"
- Git clone failure: standard git error with context

## What This Design Does NOT Cover

- `tonic update <name>` (git re-fetch) — deferred to a follow-up.
- `[[bin]]` manifest section — the `bin/` directory convention is sufficient for MVP.
- Registry source resolution — prints an actionable error directing to path/git sources.
- Windows support — follows existing Unix-first posture.
- Compiled binary optimization — shims via shell scripts are sufficient.
