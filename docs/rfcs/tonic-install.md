# RFC: `tonic install` — Global Module Installation

## Summary

Add `tonic install`, `tonic uninstall`, and `tonic installed` subcommands that let users install tonic modules globally so their binaries are available on `$PATH`. Installed modules are cached under `~/.tonic/packages/` with shell shims placed in `~/.tonic/bin/`.

## Motivation

Tonic modules like `tonic-loops` expose CLI tools via `bin/` scripts (e.g., `bin/miniloops`). Today, using these tools requires manual symlinking or invoking them by full path. There is no standard way to make a tonic module's binaries globally available.

`tonic install` formalizes the existing `bin/` convention into a first-class install workflow, analogous to `cargo install`, `deno install`, or `npm install -g`.

## Design

### CLI Surface

```
tonic install <source> [--copy] [--force]
tonic uninstall <name>
tonic installed
```

**`tonic install <source>`**

| Source form | Example | Behavior |
|---|---|---|
| Local path | `tonic install .` | Symlink into `~/.tonic/packages/<name>/` (editable). `--copy` snapshots instead. |
| Git URL | `tonic install https://github.com/user/tonic-loops.git` | Clone into `~/.tonic/packages/<name>/`. |
| Git URL + ref | `tonic install https://github.com/user/tonic-loops.git#v1.2` | Clone and checkout the specified ref. |
| Registry name | `tonic install tonic-loops` | Future — errors with "registry install not yet supported". |

Flags:
- `--copy` — copy instead of symlink for local-path installs.
- `--force` — overwrite shims owned by another package.

**`tonic uninstall <name>`** — removes shims, cached source, and manifest entry.

**`tonic installed`** — lists installed packages with source, binaries, and install date.

### Directory Layout

```
~/.tonic/
├── bin/                      # Shims (user adds to PATH)
│   ├── miniloops
│   └── my-tool
├── packages/                 # Cached project source trees
│   ├── tonic-loops/          # Symlink (local) or clone (git)
│   └── my-tool/
└── packages.toml             # Global install manifest
```

### Binary Discovery

1. Scan `bin/` directory for executable files — each becomes a shim.
2. Fallback: if no `bin/` exists but `[package] name` is set, create a single shim that runs `tonic run <package-dir>`.
3. If neither: error with actionable hint.

### Shim Strategy

Passthrough shell shims that `exec` the original `bin/<name>` script:

```sh
#!/bin/sh
# Generated by tonic install — do not edit
# Package: tonic-loops, bin: miniloops
set -eu
exec '/Users/rook/.tonic/packages/tonic-loops/bin/miniloops' "$@"
```

- `exec` replaces the shell process (no extra PID).
- Paths are quoted to handle spaces.
- Shims are `chmod +x` on creation.
- Passthrough is correct for all cases and avoids parsing original script intent.

### Global Manifest: `~/.tonic/packages.toml`

```toml
[packages.tonic-loops]
source = "path"
path = "/Users/rook/projects/tonic-loops"
symlink = true
bins = ["miniloops"]
installed_at = "2026-03-27T12:00:00Z"

[packages.other-tool]
source = "git"
url = "https://github.com/user/other-tool.git"
ref = "main"
commit = "abc1234def5678"
bins = ["other-tool"]
installed_at = "2026-03-27T14:00:00Z"
```

### Install Flow

1. Resolve source to absolute path (or clone git URL).
2. Read `tonic.toml` for package name; fall back to directory name.
3. Check for conflicts (package name or binary name collisions).
4. Create `~/.tonic/{bin,packages}/` if missing.
5. Symlink (or copy/clone) source into `~/.tonic/packages/<name>/`.
6. Run dependency sync if the package has `[dependencies]`.
7. Discover binaries from `bin/` directory.
8. Generate shims in `~/.tonic/bin/`, `chmod +x`.
9. Update `packages.toml`.
10. Print summary. On first install, print PATH setup instructions.

### Uninstall Flow

1. Look up package in `packages.toml`.
2. Remove shims from `~/.tonic/bin/`.
3. Remove `~/.tonic/packages/<name>/` (symlink or cloned tree).
4. Remove entry from `packages.toml`.
5. Print confirmation.

### PATH Setup UX

On first install, print shell-specific instructions:

```
To use installed commands, add ~/.tonic/bin to your PATH:

  # bash/zsh
  export PATH="$HOME/.tonic/bin:$PATH"

  # fish
  fish_add_path ~/.tonic/bin
```

No automatic shell profile modification.

### Conflict Resolution

- **Reinstall**: updates in place (re-symlink/re-clone, regenerate shims).
- **Binary name collision**: error with "binary 'foo' already installed by package 'bar'; use --force to overwrite".
- **`--force`**: overwrites shims and reassigns ownership in `packages.toml`.

### Error Handling

All errors use `CliDiagnostic` with actionable hints:
- Missing `tonic.toml`: "path does not appear to be a tonic project (no tonic.toml found)"
- No binaries: "no installable binaries found in <path>; add executables to bin/"
- Binary conflict: "binary '<name>' already installed by package '<pkg>'; use --force to overwrite"
- Git clone failure: standard git error with context

### Implementation Pattern

New file `src/cmd_install.rs` following the existing `cmd_*.rs` convention:
- `handle_install(args: Vec<String>) -> i32`
- `handle_uninstall(args: Vec<String>) -> i32`
- `handle_installed(args: Vec<String>) -> i32`

Dispatch in `src/main.rs`:
```rust
Some("install") => handle_install(iter.collect()),
Some("uninstall") => handle_uninstall(iter.collect()),
Some("installed") => handle_installed(iter.collect()),
```

### New Rust Types

```rust
struct InstalledPackage {
    source: InstallSource,
    bins: Vec<String>,
    installed_at: String,
}

enum InstallSource {
    Path { path: PathBuf, symlink: bool },
    Git { url: String, git_ref: Option<String>, commit: String },
}
```

## Scope Exclusions

- `tonic update <name>` (git re-fetch) — follow-up.
- `[[bin]]` manifest section — `bin/` directory convention is sufficient for MVP.
- Registry source resolution — prints actionable error.
- Windows support — follows existing Unix-first posture.
- Compiled binary optimization — shell shims are sufficient.
